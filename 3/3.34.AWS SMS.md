# 工具操作分工与SMS状态详解

## 一、工具操作细节的负责主体

**AWS工具操作细节主要由企业团队负责，而非AWS工程师**。AWS提供的是服务框架和API，具体操作由客户团队执行，这符合AWS"责任共担模式"(Shared Responsibility Model)。

### 1. 责任分工明细

| 职责范畴 | 负责方 | 具体内容 |
|---------|--------|---------|
| **服务选择与架构设计** | **企业解决方案架构师** | 根据业务需求选择合适的AWS服务组合，设计整体架构方案（SAP考试重点考察） |
| **工具配置与操作** | **企业DevOps/IT团队** | 安装配置代理、创建迁移任务、监控进度、故障排除等操作细节 |
| **服务底层运行** | **AWS官方团队** | 服务基础设施维护、安全保障、高可用性实现等"黑盒"操作 |

### 2. 团队分工模式

AWS推荐企业采用以下团队结构执行云迁移和管理：

- **应用程序团队**：负责应用代码开发和维护 
- **基础设施团队**：负责AWS资源部署与管理
- **工具团队**：负责构建和管理CI/CD及迁移管道 

**SAP考试重点考察架构师的选型决策能力，而非操作细节**，因为实际工作中：
- 架构师决定"用什么工具"（场景匹配）
- 工程师负责"怎么用工具"（操作细节）

## 二、AWS SMS(Server Migration Service)的状态

**AWS SMS已被官方正式弃用，全面推荐使用AWS MGN(Application Migration Service)替代**。

### 1. 弃用时间线

- **2022年1月1日**：停止接受新的SMS迁移任务 
- **2022年3月31日**：已启动的迁移项目必须完成 
- **2023年3月31日**：全面停止SMS服务（包括API调用） 

### 2. 弃用原因与替代方案

**AWS官方明确推荐**："Going forward, we recommend AWS Application Migration Service (AWS MGN) as the primary migration service for lift-and-shift migrations." 

**SMS vs MGN核心差异**：

| 功能特性 | AWS MGN (推荐) | AWS SMS (已弃用) |
|---------|----------------|-----------------|
| **无代理支持** | ✓ 支持无代理迁移，简化部署 | ✗ 仅支持有代理方式  |
| **大规模迁移** | ✓ 支持1000+服务器统一管理 | ✗ 适合小规模，无统一管理界面  |
| **零停机能力** | ✓ 持续增量复制，真正实现零停机 | ✗ 快照式迁移，有停机窗口  |
| **与Migration Hub集成** | ✓ 提供完整迁移可视化与管理 | ✗ 集成能力有限  |

## 三、总结

1. **工具操作分工**：
   - 企业架构师：负责"场景→工具"选型决策（SAP考试核心）
   - 企业工程师：负责工具操作细节（安装、配置、监控）
   - AWS团队：负责服务底层运行维护（不参与客户具体操作）

2. **SMS状态**：
   - **已完全退役**（2023年3月后）
   - **官方推荐使用MGN**替代，MGN提供更强大的无代理、大规模、零停机迁移能力

**SAP考试专注考察架构设计思维，而非操作手册**，这与实际云架构师工作高度一致——架构师的核心价值在于"做正确的选择"，而非"执行具体步骤"。
