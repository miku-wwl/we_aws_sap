#### 📝 [72/529] RDS 连接恢复 (Proxy)

**1. 🕵️‍♂️ 题眼与约束分析**
* **故障：** 故障转移后，应用失去连接，**必须重启**才能重连。
* **目标：** **Re-establish without restarting** (无需重启即可重连)。

**2. ⚡ 秒杀思路**
* **连接保持：** **RDS Proxy** 能够屏蔽后端的故障转移，对前端应用保持连接。
* **锁定 B。**

**3. ✅ 正确选项解析 (选项 B)**
* **RDS Proxy:** 它维护一个连接池。当 RDS 发生主备切换时，Proxy 会排队新的请求，直到新主库就绪，然后继续转发。应用层看到的只是短暂的延迟，而不是连接断开（Connection Refused），因此无需重启。

**4. ❌ 错误选项排查**
* **选项 A (Aurora Serverless v1):**
    * *错误原因：* 迁移成本高。且 v1 有冷启动问题，不适合需要高可用连接的场景。
* **选项 C (Aurora Cluster + Proxy):**
    * *错误原因：* **过度设计**。虽然 Aurora 很好，但题目只要求解决连接问题。在现有 RDS 上加 Proxy (B) 比迁移整个数据库到 Aurora (C) 要快且风险小。
* **选项 D (Athena):**
    * *错误原因：* **离谱选项**。Athena 是 Serverless 查询分析服务，用于查 S3 里的文件，根本不能替代 MySQL 做 OLTP 业务数据库。

**5. 📚 核心考点:** RDS Proxy 的故障转移透明化特性。
