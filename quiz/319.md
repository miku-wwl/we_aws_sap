#### 📝 [319/529] Amazon Managed Streaming for Apache Kafka

**1. 🕵️‍♂️ 题眼与约束分析**
* **场景：** 现有基于 Kafka 的消息系统需要云化。
* **需求：** 托管 Kafka 集群，降低运维负担。
* **约束：** 保持现有应用兼容性。
* **扩展要求：** 自动扩展，高可用性部署。

**2. ⚡ 秒杀思路**
* **Kafka 部署选项：**
    * **EC2 自建:** 完全控制但运维复杂。
    * **容器化部署:** 需要管理容器和集群。
    * **Amazon MSK:** 全托管 Kafka 服务。
* **MSK 核心特性：**
    * **100% 兼容:** 标准 Apache Kafka API。
    * **自动扩展:** 根据流量自动调整。
    * **多 AZ 部署:** 内置高可用性。
    * **安全集成:** IAM 认证，加密传输。
* **迁移优势：**
    * **选项 A (保持自建):** 继续承担运维负担。
    * **选项 B (重写应用):** 使用 SQS/SNS，开发成本高。
    * **选项 C (MSK):** 
        * 无缝应用迁移。
        * 降低运维复杂度。
        * 提升系统可靠性。
    * **选项 D (第三方云服务):** 供应商锁定风险。
* **迁移策略：**
    * 创建 MSK 集群配置。
    * 配置网络和安全组。
    * 逐步将应用连接切换到 MSK。
    * 验证功能和性能。

**3. ✅ 正确选项解析 (选项 C)**
* **Amazon MSK:** 完全托管的 Apache Kafka 服务，兼容现有应用，简化运维。

**5. 📚 核心考点:** MSK 与自建 Kafka 的对比和云原生消息架构设计。
