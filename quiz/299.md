#### 📝 [299/529] Amazon Managed Streaming for Kafka (MSK) 集群

**1. 🕵️‍♂️ 题眼与约束分析**
* **场景：** 现有应用使用 Apache Kafka。
* **需求：** 托管 Kafka 服务，简化运维。
* **约束：** 保持现有应用兼容性。
* **扩展要求：** 自动扩展，高可用性。

**2. ⚡ 秒杀思路**
* **Kafka 部署选项：**
    * **EC2 自建:** 完全控制但运维复杂。
    * **ECS/EKS:** 容器化部署，仍需管理集群。
    * **MSK:** 完全托管的 Kafka 服务。
* **MSK 核心特性：**
    * **原生 Kafka:** 100% Apache Kafka 兼容。
    * **自动扩展:** 根据流量自动调整集群大小。
    * **多 AZ:** 内置高可用性。
    * **安全集成:** IAM 认证，加密传输。
* **迁移策略：**
    * **选项 A (保持自建):** 继续承担运维负担。
    * **选项 B (重写应用):** 开发成本高。
    * **选项 C (MSK):** 
        * 无缝迁移现有应用。
        * 降低运维复杂度。
        * 提高可用性。
    * **选项 D (SQS):** 协议不兼容，需要重写。
* **实施要点：**
    * 创建 MSK 集群。
    * 配置安全组和网络。
    * 更新应用连接字符串指向 MSK 端点。

**3. ✅ 正确选项解析 (选项 C)**
* **Amazon MSK:** 托管式 Kafka 服务，兼容现有应用，降低运维成本。

**5. 📚 核心考点:** MSK 的优势、配置最佳实践与自建 Kafka 的对比。
