#### 📝 [292/529] Athena Federated Query 跨数据源查询

**1. 🕵️‍♂️ 题眼与约束分析**
* **场景：** 需要同时查询 S3 和 RDS 数据。
* **需求：** 联合查询多个数据源，统一分析。
* **约束：** 不需要数据复制，实时数据访问。
* **性能要求：** 查询延迟可接受，成本可控。

**2. ⚡ 秒杀思路**
* **跨数据源查询方案：**
    * **数据复制:** ETL 到数据湖，延迟高。
    * **Spark 联邦查询:** 需要集群管理。
    * **自定义连接器:** 开发复杂。
    * **Athena Federated Query:** 原生联邦查询能力。
* **Athena 联邦查询特性：**
    * **数据源连接器:** 支持 RDS、Redshift、MongoDB 等。
    * **查询优化:** 智能谓词下推。
    * **统一 SQL:** 标准 SQL 查询异构数据源。
    * **无服务器:** 按查询付费。
* **实施策略：**
    * **选项 A (ETL):** 数据延迟，存储成本高。
    * **选项 B (Glue ETL):** 同样需要数据复制。
    * **选项 C (Athena Federated):** 
        * 实时查询异构数据源。
        * 无需数据复制。
        * 统一分析界面。
    * **选项 D (Redshift Spectrum):** 只能查询 S3，不支持 RDS。
* **配置要点：**
    * 部署 Athena 连接器 Lambda。
    * 配置数据源连接信息。
    * 使用 Glue Catalog 统一元数据管理。

**3. ✅ 正确选项解析 (选项 C)**
* **Athena Federated Query:** 跨数据源实时联合查询，避免数据复制。

**5. 📚 核心考点:** Athena 联邦查询配置和在数据分析架构中的应用。
