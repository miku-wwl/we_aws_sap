#### 📝 [291/529] Athena 跨数据源联邦查询

**1. 🕵️‍♂️ 题眼与约束分析**
* **场景：** 需要分析存储在 S3 和 RDS 中的数据。
* **需求：** 统一 SQL 查询异构数据源，无需数据复制。
* **约束：** 查询需要实时性，数据分散在多个系统。
* **性能要求：** 查询响应时间在可接受范围内。

**2. ⚡ 秒杀思路**
* **传统数据集成方案：**
    * **ETL 复制:** 数据延迟，存储成本高。
    * **数据仓库:** 需要预先聚合数据。
    * **应用层聚合:** 复杂度高，性能差。
* **现代联邦查询：**
    * **Athena Federated Query:** 
        * 联合查询多个数据源。
        * 无需数据复制。
        * 标准 SQL 接口。
* **联邦查询优势：**
    * **实时性:** 直接查询源数据。
    * **成本效益:** 按查询付费。
    * **简化架构:** 单一查询接口。
* **实施方案：**
    * **选项 A (Glue ETL):** 需要数据复制和调度。
    * **选项 B (Redshift):** 需要数据加载。
    * **选项 C (Athena Federated):** 
        * 实时跨源查询。
        * 无数据移动。
        * 统一分析体验。
    * **选项 D (DynamoDB Global):** 仅限 NoSQL 数据。
* **技术实现：**
    * 部署 Athena 连接器。
    * 配置 RDS 连接信息。
    * 使用 Glue Catalog 统一元数据。

**3. ✅ 正确选项解析 (选项 C)**
* **Athena Federated Query:** 实现跨异构数据源的统一查询，避免数据复制。

**5. 📚 核心考点:** Athena 联邦查询在数据湖架构中的应用和配置方法。
