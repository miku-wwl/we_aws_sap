#### 📝 [303/529] AWS App Runner 容器化应用部署

**1. 🕵️‍♂️ 题眼与约束分析**
* **场景：** Web 应用需要快速部署和自动扩展。
* **需求：** 容器化部署，无需管理底层基础设施。
* **约束：** 开发团队专注于应用逻辑。
* **性能要求：** 从源码自动构建，HTTPS 终端。

**2. ⚡ 秒杀思路**
* **容器部署选项：**
    * **ECS Fargate:** 需要配置集群、服务、负载均衡器。
    * **EKS Fargate:** Kubernetes 复杂度更高。
    * **App Runner:** 完全托管的容器服务。
* **App Runner 核心特性：**
    * **源码集成:** 直接从 GitHub/CodeCommit 构建。
    * **自动扩展:** 基于请求和 CPU 自动扩展。
    * **HTTPS 默认:** 自动配置 TLS 证书。
    * **流量部署:** 支持金丝雀部署。
* **对比分析：**
    * **选项 A (ECS):** 配置复杂，学习曲线陡峭。
    * **选项 B (Elastic Beanstalk):** 主要支持传统应用，容器支持有限。
    * **选项 C (App Runner):** 
        * 完全托管体验。
        * 从源码到运行的一站式服务。
        * 自动 HTTPS 和扩展。
    * **选项 D (EC2):** 需要自己管理容器运行时。
* **部署流程：**
    * 连接代码仓库。
    * 配置构建和运行时设置。
    * App Runner 自动处理部署和扩展。

**3. ✅ 正确选项解析 (选项 C)**
* **AWS App Runner:** 面向开发者的容器应用托管服务，简化部署和运维。

**5. 📚 核心考点:** App Runner 与其他容器服务的区别及适用场景。
