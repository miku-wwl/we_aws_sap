#### 📝 [75/529] 高吞吐小数据存储 (DynamoDB TTL)

**1. 🕵️‍♂️ 题眼与约束分析**
* **特征：** 每分钟数百万条，< 4KB/条。
* **需求：** **Low latency retrieval** (低延迟检索)，保存 120 天。
* **目标：** Cost-effective。

**2. ⚡ 秒杀思路**
* **小数据高并发：** DynamoDB 完胜。
* **自动过期：** DynamoDB TTL 免费且自动。
* **锁定 B。**

**3. ✅ 正确选项解析 (选项 B)**
* **DynamoDB:** 专为 KV 型小数据设计。PutItem 效率极高。
* **TTL (Time To Live):** 这是一个后台进程，自动删除过期数据，不消耗 WCU，完全免费。

**4. ❌ 错误选项排查**
* **选项 A (S3 单文件):**
    * *错误原因：* **成本陷阱**。S3 的 `PUT` 请求费用是很贵的（$0.005/1000 requests）。每分钟数百万次 PUT 会产生天文数字的 API 费用，远超存储费用。且 S3 对海量小文件的索引和检索性能很差。
* **选项 C (RDS MySQL):**
    * *错误原因：* **性能瓶颈**。每分钟数百万写入需要极高配置的 RDS。更致命的是 `DELETE FROM table WHERE date < 120 days` 这种操作会锁表，导致数据库性能雪崩。
* **选项 D (S3 Batching):**
    * *错误原因：* **无法低延迟检索**。虽然批处理存 S3 (如 Parquet/JSON) 很便宜，但如果要查"某一条记录"，需要用 Athena 扫整个文件，延迟是秒级甚至分钟级，不符合 "Low latency retrieval"。

**5. 📚 核心考点:** 存储选型三要素：吞吐量、单条检索需求、过期策略。
