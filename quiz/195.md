#### 📝 [195/529] 游戏排行榜与高性能计算 (Redis vs DynamoDB)

**1. 🕵️‍♂️ 题眼与约束分析**
* **应用：** Node.js 游戏显示。
* **需求：** HPC 处理，**Rapidly changing leaderboard** (变化极快的排行榜)。
* **计算：** Compute optimized (计算优化)。
* **迁移策略：** Optimize performance。

**2. ⚡ 秒杀思路**
* **计算实例：**
    * 题目说 "Game application requires HPC processing"。这通常意味着需要 **Compute Optimized** 实例（C 系列）。
    * M 系列是通用型。
    * $→$ **排除 A, D** (用 m5)。
    * $→$ **候选 B, C** (用 c5)。
* **排行榜存储：**
    * **Redis (ElastiCache):** 拥有原生的 **Sorted Sets** 数据结构，是实现即时排行榜（Leaderboard）的业界标准，性能极高。
    * **OpenSearch (B):** 虽然也能排，但不如 Redis 专精且快。
    * **DynamoDB (D):** 也能做，但 Redis 是内存级的，对于 "Rapidly changing" 更合适。且 C 选项搭配了正确的 c5 实例。
    * **再看 Spot vs On-Demand:**
        * 选项 B 用 Spot。游戏平台如果是核心服务，Spot 有风险。
        * 选项 C 用 On-Demand。
        * 题目没强调"最低成本"，而是"Optimize performance"。且 c5 + Redis 是高性能的标准配置。
        * 其实 B (OpenSearch) 做排行榜是非主流的。C (Redis) 是主流。
    * **锁定 C。**

**3. ✅ 正确选项解析 (选项 C)**
* **c5 Instances:** 计算优化，适合 HPC。
* **ElastiCache Redis:** 排行榜神探 (Sorted Sets)。

**5. 📚 核心考点:** 游戏排行榜架构选型 (Redis) 与实例类型选择。
