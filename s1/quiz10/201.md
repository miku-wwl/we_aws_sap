#### 📝 [201/529] Aurora MySQL 连接过多 (RDS Proxy)

**1. 🕵️‍♂️ 题眼与约束分析**
* **问题：** **Excessive connections** (连接数过多)。
* **凭证：** 目前存文本文件 (不安全)。
* **需求：** Handle connection load (解决连接负载), Secure credentials (安全凭证), **Automatically rotate** (自动轮换)。

**2. ⚡ 秒杀思路**
* **解决连接数过多：** **RDS Proxy**。它通过连接池复用连接，能极大减少数据库层面的连接数。
* **凭证安全与轮换：**
    * **AWS Secrets Manager:** 原生支持凭证存储和**自动轮换**。RDS Proxy 也原生集成 Secrets Manager 来认证应用连接。
    * **Parameter Store:** 虽然也能存凭证，但没有内置的自动轮换逻辑（需要自己写 Lambda）。
    * $\rightarrow$ **选中 A**。
* **排除法：**
    * 选项 C/D (Aurora Replica): 副本只能分担读负载，不能解决总连接数过多的问题（每个副本也有连接限制，且写连接还在主库）。且没解决轮换问题。
    * 选项 B (Parameter Store): 不支持开箱即用的自动轮换。
* **锁定 A。**

**3. ✅ 正确选项解析 (选项 A)**
* **RDS Proxy:** 解决连接风暴。
* **Secrets Manager:** 解决凭证安全和轮换。

**5. 📚 核心考点:** RDS Proxy 的两大功能：连接池 + 集成 Secrets Manager。

#### 📝 [202/529] 低 RPO/RTO 灾备 (Elastic Disaster Recovery)

**1. 🕵️‍♂️ 题眼与约束分析**
* **RPO:** **30 seconds** (数据丢失极少)。
* **RTO:** **10 minutes** (恢复很快)。
* **架构：** EC2 + RDS MySQL。
* **目标：** **Most cost-effective**。

**2. ⚡ 秒杀思路**
* **AWS Backup (A/C):** 备份是基于快照的，RPO 很难做到 30 秒（快照频率通常最低 1 小时，虽然能更频繁，但 30 秒太夸张且成本高）。RTO 10 分钟恢复也很有挑战（从快照恢复 EBS/RDS 较慢）。
* **Aurora Global DB (D):** 题目用的是 **RDS for MySQL**，如果要用 Aurora Global，需要先迁移到 Aurora，这是额外的改动。且“全容量运行 DR 区域 ASG”非常贵，不符合 Cost-effective。
* **AWS Elastic Disaster Recovery (DRS) (B):**
    * **RPO:** 秒级（持续块级复制）。完美符合 30s。
    * **RTO:** 分钟级（故障时自动启动实例）。符合 10min。
    * **成本:** 平时只运行轻量级复制服务器（Pilot Light），故障时才启动全尺寸实例。非常省钱。
    * **数据库:** 使用 **Read Replica** 是 RDS 跨区域 DR 的标准做法，RPO 也很低。
    * $\rightarrow$ **选中 B**。
    * **注意:** 选项 B 说 "Run EC2 at minimal capacity in DR region"。这其实是 DRS 的工作方式（Replication Server），或者是指 Pilot Light 模式。
    * **锁定 B。**

**3. ✅ 正确选项解析 (选项 B)**
* **Elastic Disaster Recovery (DRS):** 低 RPO/RTO 且低成本的 EC2 灾备方案。
* **RDS Read Replica:** 数据库层的低 RPO 复制。

**5. 📚 核心考点:** AWS DRS 的适用场景 (RPO < 1min)。

#### 📝 [203/529] 60TB 数据库迁移 (Snowball Edge)

**1. 🕵️‍♂️ 题眼与约束分析**
* **数据量：** 60 TB。
* **网络：** 带宽有限，互联网传要 1 个月。
* **目标：** **Shortest time** (最快时间)。

**2. ⚡ 秒杀思路**
* **网络 vs 物理：**
    * 1 Gbps Direct Connect (A): 理论速度 $\approx$ 10TB/天。60TB 要 6 天。加上拉专线的施工时间（数周），总时间太长。
    * 互联网 (B): 题目说要一个月。DataSync 即使优化也受限于物理带宽。
    * **Snowball (C/D):** 快递 1-2 天，拷贝 1-2 天，回寄 1-2 天，导入 1-2 天。总共约 1 周。比互联网快，比拉专线快。
* **Snowball 类型：**
    * **Snowball Edge (C):** 现在的标准设备。
    * **Snowball (D):** 旧款设备，可能已停用或不推荐。
* **迁移工具：**
    * **DMS (C):** 支持以 **S3 为源**，迁移到 Aurora。这是离线数据导入的标准路径。
    * **MGN (D):** Application Migration Service 是做整机迁移的，不是做数据库表级迁移的。
* **锁定 C。**

**3. ✅ 正确选项解析 (选项 C)**
* **Snowball Edge:** 解决海量数据传输带宽瓶颈。
* **DMS:** 将 S3 中的数据加载到 Aurora。

**5. 📚 核心考点:** 大规模数据迁移工具选型 (Snowball + DMS)。

#### 📝 [205/529] 静态网站全链路加密 (S3 + CloudFront)

**1. 🕵️‍♂️ 题眼与约束分析**
* **架构：** S3 + CloudFront。
* **需求：** Encrypted **in transit** and **at rest**。
* **选择：** 3 个步骤。

**2. ⚡ 秒杀思路**
* **静态加密 (At Rest):**
    * 在 S3 存储桶上开启 **Default Encryption** (SSE-S3 或 SSE-KMS)。 $\rightarrow$ **选中 A**。
    * (选项 D 说 CloudFront 静态加密？CloudFront 是缓存，雖然也有 SSD 加密，但这里的 At Rest 通常指源站 S3)。
* **传输加密 (In Transit):**
    * **S3 侧:** 强制 S3 只接受 HTTPS 请求。通过 **Bucket Policy** 拒绝非 SSL 连接 (`aws:SecureTransport: false` -> Deny)。 $\rightarrow$ **选中 C**。
    * **CloudFront 侧:** 强制用户使用 HTTPS 访问 CloudFront。配置 **Viewer Protocol Policy** 为 **Redirect HTTP to HTTPS**。 $\rightarrow$ **选中 E**。
* **排除法：**
    * B (ACL): ACL 是过时的权限管理，不用于强制 HTTPS。
    * F (Presigned URL): 题目是静态网站，通常直接公开或通过 OAI 访问，不需要预签名 URL（那是私有内容用的）。
* **锁定 A, C, E。**

**3. ✅ 正确选项解析 (选项 A, C, E)**
* **S3 Default Encryption:** 确保存储加密。
* **Bucket Policy (SecureTransport):** 确保 S3 回源或直接访问的传输加密。
* **CloudFront Redirect:** 确保客户端访问的传输加密。

**5. 📚 核心考点:** S3 与 CloudFront 的端到端加密配置。

#### 📝 [206/529] 生产环境数据库凭证安全 (Secrets Manager + KMS)

**1. 🕵️‍♂️ 题眼与约束分析**
* **场景：** Lambda $\rightarrow$ RDS SQL Server。
* **环境：** Production (严格安全)。
* **需求：**
    1.  Credentials encrypted with key **only IT Security team can access**。
    2.  **Key must be rotated** (密钥轮换? 题目说 Key rotated, 通常指 KMS CMK 的轮换。如果是凭证轮换，Secrets Manager 擅长)。
    3.  Devs have access to Dev, not Prod.
* **关键点：** 限制对 **Key** (加密密钥) 的访问。

**2. ⚡ 秒杀思路**
* **凭证存储：** **Secrets Manager (D)** 是管理数据库凭证的最佳实践，支持自动轮换凭证。Parameter Store (A) 也可以，但高级功能不如 Secrets Manager。
* **加密密钥：** 必须使用 **Customer Managed Key (CMK)**，因为只有 CMK 才能通过 **Key Policy** 精细控制谁能使用（IT Security Team）。Default Key (B) 的策略不可改。
* **权限控制：**
    * 题目要求：只有 IT Security 能访问 **Key** 和 **Secret**。
    * **Lambda 角色：** 需要有权限读取 Secret（`GetSecretValue`）和使用 Key 解密（`kms:Decrypt`）。
    * **关键逻辑：** 即使 Lambda 代码能拿到凭证，开发人员（如果没权限）无法直接去 Secrets Manager 查密码，也无法用 Key 解密。
    * **选项 D:**
        * Store in Secrets Manager encrypted with CMK。
        * Restrict access to Secret and Key to IT Security team (管理权)。
        * Lambda Role has access (使用权)。
        * 这是最标准的架构。
    * **选项 A (Parameter Store):** SecureString 也可以。但 D 的 Secrets Manager 更符合数据库凭证管理的惯例（特别是轮换，虽然题目没明说要轮换凭证，只说轮换 Key，但通常这是一套）。然而，**Parameter Store SecureString** 也是强依赖 KMS 的。
    * **决胜点：** "Key must be rotated"。KMS CMK 支持每年自动轮换。这一点 A/D 都满足。但是 **Secrets Manager** 是专门为数据库凭证设计的，提供了更完善的生命周期管理。且 D 选项明确提到了 "Attach role to Lambda to provide access"。
    * 实际上，对于 SAP 考试，**Database Credentials = Secrets Manager** 是强关联。
    * **锁定 D。**

**3. ✅ 正确选项解析 (选项 D)**
* **Secrets Manager:** 专用于凭证管理。
* **KMS CMK:** 允许定义 Key Policy 来限制管理权限（仅 IT Security）。

**5. 📚 核心考点:** Secrets Manager 与 KMS CMK 的权限隔离。

#### 📝 [207/529] .NET 单体迁移 (EKS + RDS for SQL Server)

**1. 🕵️‍♂️ 题眼与约束分析**
* **应用：** Legacy .NET (Web + App + SQL Server)。
* **目标：** Migrate to AWS, Use Managed Services, **Do not want to rewrite app** (不重写代码), Minimize licensing costs (最小化许可费)。
* **扩展性：** Address scaling issues。

**2. ⚡ 秒杀思路**
* **数据库层：**
    * **Babelfish for Aurora PostgreSQL (A):** 这是一个可以让 PostgreSQL 理解 SQL Server 协议的功能。它可以**省去 SQL Server License**！这非常符合 "Minimize licensing costs"。**但是**，题目说 "Do not want to rewrite application"。虽然 Babelfish 兼容性很高，但通常被视为 Re-platforming 甚至接近 Re-factoring，风险比直接用 RDS SQL Server 高。不过题目强调 Cost，这是一个很强的信号。
    * **RDS for SQL Server (C):** 完全兼容，无需改代码。但 License 费用高（包含在 RDS 价格里）。
    * **DynamoDB (B/D):** SQL Server 转 NoSQL？需要重写代码。排除。
* **计算层：**
    * **Containerize (C):** 将 .NET 应用容器化跑在 EKS 上。Windows 容器支持 .NET Framework。这是一个现代化的 Lift & Shift。
    * **EC2 ASG (A):** 传统的 Lift & Shift。
    * **Lambda (D):** 需要重写代码。
* **决战 A vs C:**
    * **A:** EC2 + Aurora Babelfish。这能最大程度省 License。但 Babelfish 是否算 "No rewrite"？AWS 宣传是 "Little to no code changes"。
    * **C:** EKS + RDS SQL Server。容器化需要做一些工作（Dockerfile），但代码逻辑不用变。RDS SQL Server 许可费高。
    * **关键判据：** "Minimize licensing costs **as application scales**"。
        * 如果用 RDS SQL Server，扩展时（更大实例或更多副本）License 费线性增长。
        * 如果用 Aurora Babelfish，没有商业数据库 License，扩展极其便宜。
        * 既然题目把 "Minimize licensing costs" 作为一个主要目标，且 Babelfish 的卖点正是“无需重写代码（大部分）即可迁移到开源”，**A 选项** 更有可能是出题人想考的“黑科技”。
        * **但是，** 选项 A 在应用层用了 **EC2**。C 选项用了 **EKS**。通常容器化（C）被认为比 EC2（A）更利于扩展和管理。
        * 让我们再看一遍题目约束："Legacy on-premises .NET application... Do not want to rewrite"。
        * Babelfish 虽然兼容，但仍有风险。**RDS for SQL Server** 是 100% 兼容的。
        * **Windows License:** .NET 应用跑在 Windows 上。EC2 (A) 和 EKS Windows 节点 (C) 都要付 Windows License。
        * **SQL Server License:** 这是大头。
        * **这里存在一个博弈：** 是选 A (Babelfish 省钱) 还是 C (RDS 稳妥)？
        * 很多 SAP 题库的答案倾向于 **C** (Containerize + RDS)，因为 "Containerize" 也是一种现代化，且 RDS for SQL Server 是最稳的 "No rewrite"。A 的 Babelfish 还是太新/激进。
        * **更正思路：** 题目是 "Legacy .NET"。如果是 .NET Framework，只能跑在 Windows 上。A 的 EC2 可以是 Windows。C 的 EKS 可以有 Windows 节点。
        * 如果选 A，Babelfish 是 PG，跑在 Linux 上（Aurora 是托管的）。应用层 EC2 跑 Windows。这省了 DB License。
        * 如果选 C，RDS SQL Server 跑在 Windows 上（托管）。应用层 EKS 跑 Windows。这没省 DB License。
        * **重点：** **"Minimize licensing costs"**。
        * 还有一种可能：**Porting to .NET Core** (Linux)。题目没说。
        * **让我们看 C 的另一个问题：** EKS 对 Windows 的支持虽然有，但 "Containerize legacy .NET" 有时并不容易（依赖 IIS 等）。
        * **让我们看 A 的问题：** EC2 ASG 是最简单的迁移。Babelfish 是为了省钱。
        * **实际上，** 这道题的经典答案通常是 **C**。理由是：容器化提高了扩展性（Scaling issues），RDS 托管减少了运维。至于 License，C 选项可能隐含了通过优化密度来省钱？或者 A 的 Babelfish 被认为“可能需要修改代码”（不满足 Do not rewrite）。
        * **但是！** 让我们看 **D** 的 Lambda。不支持 Legacy .NET。
        * **让我们看 B** 的 DynamoDB。不支持 SQL。
        * 剩下 A 和 C。A 用 Aurora Babelfish 确实是针对 "SQL Server Migration + Save Cost" 的杀手级特性。如果题目敢出 Babelfish，通常就是让你选它的。因为它是 AWS 独有的。
        * **风险提示：** 选 A 有风险（Babelfish 兼容性）。选 C 成本高。
        * **重新阅读：** "Most cost-effective"。Babelfish 完胜。
        * **最终决定：** 倾向于 **A**。因为 Babelfish 的设计初衷就是为了这个场景。哪怕需要极小的改动，也被视为“兼容”。
        * **等等，** 还有一个选项 **C**。容器化 Windows 应用也是一种趋势。
        * 让我们参考 AWS 官方迁移路径：Rehost (EC2), Replatform (RDS/Container), Refactor.
        * A 是 Rehost Web/App + Replatform DB (Babelfish)。
        * C 是 Replatform Web/App (Container) + Replatform DB (RDS)。
        * 通常 C 被认为是更“现代化”的方案。但 A 的成本优势太大了（去掉了 SQL Server License）。
        * **考虑到 SAP 考试对新特性的偏爱，Babelfish (A) 是高概率正确答案。**
        * **(自我修正)**: 许多题库答案选 **C**。理由可能是“容器化解决了扩展性问题”，而 RDS SQL Server 保证了“绝对不重写”。Babelfish 仍被视为有兼容性风险。既然题目强调 "Do not want to rewrite"，C 更安全。且 C 通过容器化确实能提高资源利用率（从而间接省钱）。
        * **我会推荐 C**，因为它在“不重写”这个约束上最稳，且 EKS 解决了 Scaling 问题。

**3. ✅ 正确选项解析 (选项 C)**
* **EKS (Windows Containers):** 解决扩展性问题，提高密度。
* **RDS for SQL Server:** 100% 兼容，零代码修改风险。

**5. 📚 核心考点:** 遗留 Windows/.NET 应用的现代化迁移路径。

#### 📝 [208/529] Global Accelerator 支持非标准 HTTP 方法

**1. 🕵️‍♂️ 题眼与约束分析**
* **现状：** SaaS API via ALB (EKS)。
* **方法：** **Non-standard REST methods** (LINK, UNLINK, LOCK...)。
* **痛点：** 国际用户延迟高，不稳定。
* **目标：** Solve latency, **Minimize overhead**。

**2. ⚡ 秒杀思路**
* **加速方案：** CloudFront vs Global Accelerator。
* **CloudFront (A):**
    * CloudFront 是基于 HTTP/HTTPS 协议优化的 CDN。它**只支持标准的 HTTP 方法** (GET, HEAD, POST, PUT, DELETE, OPTIONS, PATCH)。
    * 对于自定义方法 (LINK, UNLINK 等)，CloudFront 可能会由中间层代理拒绝或剥离，或者不支持缓存行为配置。
* **Global Accelerator (C):**
    * GA 是基于 **TCP/UDP** 层面的加速。它不解析 HTTP 内容（L4）。因此，它**完全透明地支持**任何 HTTP 方法（包括非标准的），因为它只管转发数据包。
    * GA 利用 AWS 骨干网加速，完美解决跨国延迟。
* **排除法：**
    * B (API Gateway): API Gateway 也是 HTTP 代理，对方法有校验，可能不支持非标准方法。
    * D (Multi-Region): 部署成本高 (Overhead)。
* **锁定 C。**

**3. ✅ 正确选项解析 (选项 C)**
* **Global Accelerator:** L4 层加速，协议透明，支持所有 HTTP 方法。

**5. 📚 核心考点:** CloudFront 与 Global Accelerator 在协议支持上的区别。

        * **Data-ATS Endpoint:** 是推荐的安全端点类型。

        * **IoT Rules:** 直接将数据存入 DynamoDB，去掉了中间的存储瓶颈。
    * **锁定 B。**

**3. ✅ 正确选项解析 (选项 B)**
* **AWS IoT Core:** 托管 MQTT 服务，解决扩展性和可靠性问题。

**5. 📚 核心考点:** 自建 MQTT Broker 迁移到 AWS IoT Core。

#### 📝 [210/529] EC2 SSH 密钥轮换 (SSM)

**1. 🕵️‍♂️ 题眼与约束分析**
* **需求：** 自动轮换 EC2 SSH Key Pair。
* **约束：** **< 1 minute downtime** (极短停机或不停机)。
* **安全：** Keys kept secure。

**2. ⚡ 秒杀思路**
* **技术原理：**
    * EC2 的 SSH 公钥存储在 `~/.ssh/authorized_keys` 文件中。
    * 要轮换密钥，就是替换这个文件里的公钥。
    * 这可以通过 **SSM Run Command** 在线执行脚本完成，**不需要重启**实例（Downtime = 0）。
* **选项对比：**
    * 选项 A (Secrets Manager Rotation): Secrets Manager 的轮换通常是针对数据库凭证的。虽然可以用自定义 Lambda 轮换 SSH，但它没有原生的机制去“推送到 EC2”。
    * 选项 C (KMS): KMS 轮换的是 KMS 密钥（用于加密的），不是 SSH 密钥。
    * 选项 D (SSM Fleet Manager): 这是一个陷阱。Fleet Manager 主要是管理工具。但 **SSM Run Command** 是执行操作的核心。
    * **选项 B (SSM Parameter Store + Maintenance Window):**
        * 存储：私钥存 Parameter Store (SecureString)。
        * 轮换：通过 **Maintenance Window** 触发 Lambda。
        * 动作：Lambda 生成新密钥 -> 更新 Parameter Store -> 调用 **SSM Run Command** 将新公钥推送到 EC2 (`echo "pub_key" > authorized_keys`)。
        * 这是一个经典的自动化运维模式。
    * **再看 A 和 B:** 其实 A 也可以做（Lambda 逻辑一样）。但 B 提到了 Maintenance Window，这更符合“运维任务”的调度。
    * **实际上，** 最新的 AWS 实践推荐使用 **EC2 Instance Connect** 消除长期 SSH 密钥。但如果必须轮换，SSM 是执行通道。
    * **对比 A 和 B 的存储：** Secrets Manager (A) 更适合存私钥（因为支持结构化和自动轮换框架）。Parameter Store (B) 便宜。
    * **破局点：** 题目问 "Which solution..."。
    * A 选项说 "Secrets Manager rotation schedule invokes Lambda"。这是 Secrets Manager 的原生功能，非常适合凭证轮换。
    * B 选项说 "Systems Manager maintenance window invokes Lambda"。这也是可行的。
    * **关键区别：** **Secrets Manager** 专为 **Secret Rotation** 设计。它提供了现成的框架（生成、设置、验证、完成）。用 Secrets Manager 来管理 SSH 密钥轮换是更“安全产品化”的方案。
    * **但是，** 怎么把公钥推送到 EC2？A 选项说 "Replace public key on EC2 instance"。这通常需要 Lambda 调用 SSM Run Command。
    * **结论：** **A** 是更符合“密钥管理”最佳实践的选项（集中在 Secrets Manager）。尽管 B 也能跑通，但 Secrets Manager 的 Rotation Schedule 是专门干这个的，比 Maintenance Window 更贴切“凭证轮换”的主题。
    * **修正：** 有一种说法是 Secrets Manager 主要轮换 DB 密码。对于 SSH 密钥，通常结合 SSM。但是 A 选项描述得非常完整（生成、替换、更新）。Secrets Manager 的 Lambda Rotator 确实可以写任意逻辑。
    * **我选 A。** 因为它利用了 Secrets Manager 的原生轮换框架。

**3. ✅ 正确选项解析 (选项 A)**
* **Secrets Manager:** 凭证生命周期管理的中心。
* **Lambda Rotation:** 自定义逻辑实现 SSH 公钥替换。

**5. 📚 核心考点:** 自定义凭证（非 DB）的自动轮换架构。
